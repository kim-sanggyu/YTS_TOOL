# 세액계산 종합진단 — 해설서 설계 문서

> 작성일: 2026-07-02  
> 목적: 유지보수 개발자가 해설서 기능의 설계 의도와 원칙을 이해할 수 있도록 기록

---

## 1. 핵심 원칙

### CALC_PROC_TOTAL 단일 소스 원칙

**해설서의 모든 내용은 CALC_PROC_TOTAL 정보만으로 구현한다.**

이 원칙을 채택한 이유:
- 이 서비스는 고객에게 직접 제공되는 프로그램이다. 버그 발생 시 추적이 가능해야 한다.
- "이 해설이 왜 나왔지?" → CALC_PROC_TOTAL을 보면 바로 확인할 수 있어야 한다.
- DB 외부 컬럼(PAY_WRK_CALC, PAY_WRK_MAIN 등) 의존 시 문제:
  - 세법 개정 → 컬럼 추가/변경 → 해설 로직도 함께 수정 필요
  - 어떤 컬럼이 어떤 해설에 영향을 주는지 파악 어려움
  - 매년 검증 부담

### 해설에 필요한 데이터가 CALC_PROC_TOTAL에 없는 경우

두 가지 선택만 있다:

1. **꼭 필요한 기능이면** → 백엔드 계산 엔진이 CALC_PROC_TOTAL에 해당 데이터를 추가
2. **없어도 되는 기능이면** → 해설 항목 자체를 삭제

## 2. CALC_PROC_TOTAL 구조

```
① 총급여 산출
② 근로소득금액 계산

▶ 소득공제(종합소득공제)
  (잔액) N - 공제금액 (항목명) [설명]
  ...

③ 차감소득금액 계산

▶ 소득공제(그밖의소득공제)
  (잔액) N - 공제금액 (항목명) [설명]
  ...

④ 산출세액 계산
  ㆍN (산출세액) = ...

▶ 세액감면
▶ 세액공제
  (잔액) N - 공제금액 (항목명) [설명]
  ...

⑤ 결정세액 계산

[상세공제내역]         ← 파싱 사용 안 함 (아래 참고)

※ (자동/선택)특별소득ㆍ세액공제 적용 세액 N (표준적용時 N), 소진지점: 항목명
※ 계산일시
※ 성명님(사번, CALC_NO)
```

### 주요 패턴

**표준세액공제 방식 블록:**
```
'표준세액공제' 방식으로 계산합니다.
 특별소득공제, 특별세액공제※, 월세액 항목은 공제하지 않습니다.
```
→ 이후 `→ 월세액, 특례기부금` 등 미적용 항목 열거

**세액 소진 블록:**
```
▣▣▣ [교육비] 항목에서 산출세액이 모두 소진되었습니다.
     이후 항목은 더 이상 공제하지 않습니다.
▣▣▣
```

**소진지점 (하단 ※ 라인):**
```
※ (자동)특별소득ㆍ세액공제 적용 세액 0 (표준적용時 N), 소진지점: 교육비
```

## 3. 소득자 입력 값 전용 섹션

### 왜 별도 섹션이 필요한가

CALC_PROC_TOTAL 본문은 계산 결과(잔액, 공제금액)를 보여주지만,
소득자가 **입력한 원래 금액**은 일부 항목에서만, 불규칙하게 표시된다.

또한 `[상세공제내역]` 블록은:
- 구조가 복잡해 파싱 오류 발생 가능성이 높음
- 미래 개발자가 해설서 의존성을 모르고 제거할 수 있음

따라서 해설에 필요한 입력값은 **전용 섹션**으로 명시적으로 삽입한다.

### 형식

백엔드 계산 엔진이 CALC_PROC_TOTAL 내에 아래 섹션을 삽입:

```
====소득자 입력 값====
공제신고서제출여부: N
월세액: 4920000
건강보험료: 2566920
고용보험료: 590640
보장성보험료: 622620
교육비: 0
주택마련저축: 840000
```

- 입력이 없는 항목은 줄 자체를 생략 (파서는 줄 없으면 0으로 처리)
- 항목이 추가될 때마다 이 섹션에 한 줄씩 추가
- 숫자는 쉼표 없는 정수 (파싱 단순화)
- `공제신고서제출여부`만 Y/N 값, 나머지는 모두 금액(정수)

### 확정 항목 목록 (2026-07-02 기준)

- 0인 항목은 생략 (파서는 줄 없으면 0으로 처리)
- 공제신고서제출여부는 N일 때만 표시 (Y이면 생략)
- 신용카드·의료비는 CALC_PROC_CARD/MEDI JSON 컬럼으로 처리 → 전용 섹션 불필요

| 항목 | 값 형식 | 생략 안 되는 이유 |
|------|--------|-----------------|
| ~~공제신고서제출여부~~ | — | CALC_PROC_TOTAL은 계산 시점 스냅샷이라 항상 N. 제출 후 Y로 바뀌는 최신값은 PAY_WRK_MAIN.CONF_YN에서 직접 읽어야 함. 예외적으로 DB 컬럼 사용. |
| 월세액 | 정수 | 표준방식·소진 케이스에서 본문에 입력금액 없음 |
| 건강보험료 | 정수 | 표준방식에서 해당 라인 생략 |
| 고용보험료 | 정수 | 표준방식에서 해당 라인 생략 |
| 보장성보험료 | 정수 | 표준방식에서 해당 라인 생략 |
| 교육비 | 정수 | 표준방식에서 해당 라인 생략 |
| 청약저축 | 정수 | 세대원 케이스에서 입력금액 본문에 없음 |
| 주택청약종합저축 | 정수 | 동일 |
| 근로자주택마련저축 | 정수 | 동일 |
| 주택임차차입금_대출기관 | 정수 | 표준방식에서 해당 라인 생략 |
| 주택임차차입금_거주자 | 정수 | 표준방식에서 해당 라인 생략 |

## 4. 대시보드와 해설서의 구분

| | 대시보드 (`/tools/tax-dashboard`) | 해설서 (`/tools/tax-insight`) |
|--|----------------------------------|------------------------------|
| 목적 | 전체 인원 통계 집계 | 개별 건 계산 과정 해설 |
| 데이터 소스 | DB 컬럼 (SQL 집계) | CALC_PROC_TOTAL 파싱 |
| 이유 | SQL에서 직접 COUNT → 효율적 | 텍스트 파싱으로 맥락 설명 가능 |

**대시보드는 DB 컬럼을 계속 사용한다.** 전 건의 CALC_PROC_TOTAL을 앱에서 파싱하면 비효율적이고, 통계 집계에는 SQL이 적합하다. CALC_PROC_TOTAL 단일 소스 원칙은 해설서에만 적용된다.

## 5. [상세공제내역] 사용 금지

`[상세공제내역]` 블록(신용카드 상세, 의료비 상세 등)은 **파싱에 사용하지 않는다.**

단, `CALC_PROC_CARD`, `CALC_PROC_MEDI`는 별도 JSON 컬럼으로 이미 구조화되어 있어 계속 사용 가능.
(이 둘은 CALC_PROC_TOTAL과 별개 컬럼이며, 형식이 안정적이다.)

## 5. 항목별 점검 기록

각 해설 항목을 하나씩 점검하여 CALC_PROC_TOTAL 기반으로 전환하거나 삭제한다.

### 월세액 (진행 중)

**현재 코드의 외부 컬럼 의존:**
- `MAIN_HOUSE_RENT` (PAY_WRK_MAIN) — 월세 입력금액
- `RT_HOUSE_RENT_AMT` (PAY_WRK_CALC) — 공제금액
- `PROD_TAX_AMT`, `RES_INCM_TAX` (PAY_WRK_CALC) — 소진 판단
- `CALC_METHOD` — 소진지점

**CALC_PROC_TOTAL에서 파싱 가능한 것:**
- 표준방식 여부 → `'표준세액공제' 방식` 블록 + `→ 월세액`
- 세액소진 여부·소진항목 → `▣▣▣` 블록 또는 하단 `소진지점:` 라인
- 산출세액 → `④ ㆍN (산출세액)`

**전용 섹션에 추가 필요:**
- `월세액: N` — 월세 입력금액

**케이스별 해설:**
| 케이스 | 감지 조건 | 해설 |
|--------|----------|------|
| 정상 공제 | 전용섹션 월세액 > 0, 본문 공제금액 > 0 | 해설 불필요 |
| 표준방식 미적용 | 전용섹션 월세액 > 0, 표준방식 블록에 `→ 월세액` | "표준세액공제 방식 선택으로 월세 공제 미적용" |
| 세액소진 미적용 | 전용섹션 월세액 > 0, `▣▣▣` 소진 블록 | "[소진항목] 공제가 산출세액을 먼저 소진하여 월세 공제 불가" |

**상태: 백엔드 `====소득자 입력 값====` 섹션에 `월세액: N` 추가 대기 중**
